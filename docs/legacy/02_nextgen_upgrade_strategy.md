# SINWOO 차세대 업그레이드 전략

## 1. 방향 정의

이번 작업은 단순 리팩터링이 아니다.

- 레거시 유지 보수도 아님
- 빅뱅 재개발도 아님
- **기존 기능 자산을 살린 차세대 이행 업그레이드**임

즉, 현재 레거시 프로젝트의 업무 자산을 기반으로
다음 세대 운영 플랫폼으로 단계 전환한다.

## 2. 목표 상태

### Backend

- Spring Boot 3.x
- Java 21
- Gradle
- MariaDB
- Flyway
- MyBatis 중심 + 필요 구간 JPA 보조
- 공통 감사 컬럼 표준화
- 테이블별 `_HIST` + DB Trigger 기반 이력

### Frontend

- 신규 화면은 `Next.js + Tailwind CSS + shadcn/ui`
- 기존 화면은 당분간 Thymeleaf 유지 가능
- 점진적 화면 전환 전략 채택

### Platform Standard

- 표준 테이블명: `TB_*`
- 표준 인덱스명: `IX_*`, `UK_*`, `FK_*`
- 약어 기반 컬럼명
- 공통 감사 컬럼:
  - `CRT_BY`
  - `CRT_DTM`
  - `UPD_BY`
  - `UPD_DTM`

## 3. 왜 빅뱅 전환이 아닌가

레거시 프로젝트는 이미 다음 기능을 보유하고 있다.

- 회계
- 급여
- 자산
- 근태
- 개인청구
- 출장
- 보고서
- 배치
- OCR/Textract

즉 “아무것도 없는 프로젝트”가 아니라
비즈니스 자산이 이미 충분히 쌓여 있기 때문에,
전면 폐기보다 단계 이행이 훨씬 유리하다.

## 4. 차세대 이행 원칙

### 4.1 Preserve First

동작 중인 업무 흐름은 먼저 보존한다.

### 4.2 Standardize Around It

표준은 기존 기능 위에 덧씌운다.

### 4.3 Domain by Domain

한 번에 전체 전환하지 않고 도메인 단위로 옮긴다.

### 4.4 Dual Run Allowed

기존 Thymeleaf + 신규 Next.js 화면이 일정 기간 공존해도 된다.

### 4.5 DB is Contract

DB 표준화는 중요하지만, 기능을 깨는 방식으로 한 번에 교체하지 않는다.

## 5. 권장 아키텍처

## 5.1 Strangler Pattern

가장 적합한 전략은 Strangler 방식이다.

- 기존 기능은 레거시에 유지
- 신규/개선 기능은 차세대 구조에 포팅
- 검증된 뒤 도메인별로 레거시 축소

## 5.2 공통축 우선

가장 먼저 옮겨야 하는 것은 화면보다 공통축이다.

1. 인증/권한
2. 회사/부서/직원
3. 공통 코드/로케일
4. 감사/이력
5. 파일/OCR 공통 인프라

## 5.3 MyBatis 유지 전략

차세대라고 해서 모든 SQL을 JPA로 바꾸는 것은 비효율적이다.

- 회계/집계/마감/정산: MyBatis 유지
- 단순 마스터/권한/설정: JPA 또는 MyBatis 선택

즉 차세대 표준은 “JPA 올인”이 아니라,
**MyBatis를 전략적으로 계속 활용하는 구조**가 맞다.

## 6. 단계별 실행안

### Phase 0. 분석 기준 고정

- 레거시 인벤토리
- 도메인 매핑
- 핵심 테이블 식별
- 공통축 추출

### Phase 1. 플랫폼 표준 구축

- DB 표준명
- 감사 컬럼
- `_HIST` + Trigger
- 공통 에러 처리
- 공통 응답 규격
- 환경 분리(local/dev/prod)

### Phase 2. 공통축 포팅

- 사용자
- 권한
- 회사
- 부서
- 로그인/세션/보안

### Phase 3. 업무 도메인 포팅

우선순위:

1. 근태/휴가
2. 경비/출장/개인청구
3. 급여
4. 자산/마감/재무보고
5. OCR/문서관리

### Phase 4. 프론트 차세대 전환

- 신규 대시보드
- 관리자 콘솔
- 도메인별 신규 화면
- 기존 화면 점진 제거

## 7. 금지사항

- 기존 기능 검증 없이 전면 테이블명 변경
- SQL 매퍼 일괄 삭제 후 JPA 치환
- 기존 화면 전체 일괄 폐기
- 도메인 구분 없이 “한 번에 새 프로젝트”

## 8. 이번 업그레이드의 핵심 성공 조건

- 기존 기능은 계속 살아 있어야 한다.
- 도메인별 전환 후 회귀 검증이 가능해야 한다.
- 문서와 코드 표준이 같이 움직여야 한다.
- 레거시와 차세대 사이의 연결 고리가 명확해야 한다.

## 9. 결론

이번 프로젝트는 “레거시를 정리하는 작업”이 아니라,
**레거시를 기반으로 차세대 플랫폼으로 진화시키는 작업**으로 정의한다.

즉, 목표는 유지보수가 아니라 제품화다.
