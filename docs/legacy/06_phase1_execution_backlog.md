# SINWOO 차세대 업그레이드 Phase 1 백로그

## 1. 목표

Phase 1의 목적은 레거시 기능을 깨지 않으면서,
차세대 플랫폼의 공통축을 먼저 세우는 것이다.

이번 단계의 산출물은 “신규 기능”보다 다음에 가깝다.

- 표준
- 브리지
- 공통 마스터
- 회귀 가능한 이행 기반

## 2. 최우선 작업

### P0-1. 사용자/회사/권한 공통 모델 확정

목표:

- `TB_TENANT`
- `TB_CO`
- `TB_USR`
- `TB_ROLE`
- `TB_USR_ROLE`

차세대 기준으로 공통축 확정

완료 기준:

- 레거시 `tb_user`와 차세대 `TB_USR` 간 매핑 초안 승인
- 회사/부서/권한 참조 구조 결정

### P0-2. 접근 로그와 변경이력 분리 설계

목표:

- 레거시 `tb_log_history`는 운영 로그로 분리
- 데이터 변경이력은 `_HIST + Trigger`로 유지

완료 기준:

- `TB_ACCESS_LOG` 표준 초안
- 운영 로그 저장 컬럼 정의

### P0-3. 환경설정 표준화

목표:

- local/dev/prod 설정 분리
- 비밀정보 분리
- 실행 기준 문서화

완료 기준:

- 레거시 프로젝트 설정 분리안 정리
- 현행/차세대 실행 경로 명세

## 3. 1차 포팅 대상

### P1-1. 로그인/권한

이유:

- 모든 화면과 도메인의 시작점
- 공통축이므로 재사용 범위가 가장 넓음

대상:

- `WebSecurityConfig`
- `LoginMapper`
- 로그인 관련 Controller/VO
- 세션/권한 처리

### P1-2. 사용자/회사/부서

이유:

- 근태, 급여, 요청, 재무 모두의 공통 마스터

대상:

- 사용자
- 회사
- 부서
- 직원
- 프로필

### P1-3. 공통코드/로케일

이유:

- 다국어, 상태코드, 유형코드의 기준 축

## 4. 2차 포팅 대상

### P2-1. 근태/휴가

이유:

- 비교적 독립성이 높고 차세대 UX 전환 효과가 큼

### P2-2. 경비/출장/개인청구

이유:

- 화면 흐름이 명확하고 사용자 체감 가치가 높음

## 5. 3차 포팅 대상

### P3-1. 급여

이유:

- 중요도가 높지만 SQL/비즈니스 규칙이 복잡함

### P3-2. 자산/마감/재무리포트

이유:

- 회귀 리스크가 가장 높음
- 늦게 옮기는 것이 안전

## 6. 착수 순서

1. 공통축 스키마/필드 정리
2. 로그인/권한 차세대 모델 정리
3. 사용자/회사/부서 포팅
4. 접근로그 표준화
5. 근태 도메인 포팅 착수

## 7. 바로 다음 턴에서 할 일

다음 실제 개발 턴에서는 아래 둘 중 하나부터 바로 시작하면 된다.

### 선택지 A. 사용자/권한 공통축 포팅

- 공통 엔티티/테이블/DTO 기준 정리
- 레거시 로그인 흐름과 매핑

### 선택지 B. 레거시 운영 로그 표준 테이블 설계

- `tb_log_history`를 차세대 `TB_ACCESS_LOG`로 승격
- 요청/사용자/엔드포인트 추적 표준 확정

현재 추천은 **선택지 A**다.
