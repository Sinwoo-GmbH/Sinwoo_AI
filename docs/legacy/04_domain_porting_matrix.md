# SINWOO 도메인 포팅 매트릭스

## 1. 기준

포팅 우선순위는 아래 4가지를 기준으로 정한다.

- 공통성
- 변경 파급도
- 차세대 표준 필요성
- 프론트 전환 가능성

## 2. 포팅 우선순위 매트릭스

| 도메인 | 현재 자산 | 차세대 우선순위 | 접근 방식 | 비고 |
| --- | --- | --- | --- | --- |
| 로그인/권한 | 높음 | 최상 | 먼저 공통축으로 재구성 | 레거시 세션 기반을 표준 인증 구조로 승격 |
| 사용자/회사/부서 | 높음 | 최상 | 공통 마스터 우선 포팅 | 이후 모든 도메인의 기준축 |
| 공통코드/로케일 | 높음 | 최상 | 표준명과 다국어 기준 정리 | 전 도메인 공통 |
| 이력/감사/로그 | 중간 | 최상 | 플랫폼 레벨 공통화 | `_HIST` + access log 이원화 |
| 근태/휴가 | 높음 | 상 | 도메인 단위 이행 | 차세대 화면 전환 효과 큼 |
| 개인청구/출장 | 높음 | 상 | 요청/청구 흐름 우선 정리 | 사용자 경험 개선 포인트 큼 |
| 급여 | 높음 | 상 | SQL 보존 + 구조 정리 | 계산/월단위 처리 검증 중요 |
| 자산 | 높음 | 중상 | SQL 중심 유지 | 재무 마스터와 결합도 높음 |
| 마감/재무리포트 | 높음 | 중상 | 가장 늦게 포팅 | 회귀 리스크 큼 |
| OCR/문서 | 중간 | 중상 | 공통 문서 모듈로 승격 | 차세대 플랫폼 차별점 |
| 관리자 화면 | 높음 | 중상 | 신규 프론트로 점진 전환 | Next.js 우선 후보 |

## 3. 권장 포팅 순서

### Wave 1. 공통 기반

- 로그인/권한
- 사용자
- 회사
- 부서
- 공통코드
- 로케일

### Wave 2. 업무 공통

- 근태
- 휴가
- 개인청구
- 출장

### Wave 3. 재무 핵심

- 급여
- 자산
- 운영비/매출/매입

### Wave 4. 고난도 정산

- 마감
- 재무리포트
- 배치 자동화

### Wave 5. 제품화 강화

- OCR/문서관리
- 차세대 대시보드
- 관리자 콘솔

## 4. 포팅 원칙

### Keep

- 업무 규칙
- 핵심 SQL 의미
- 도메인 용어
- 화면 흐름

### Refactor

- 패키지 구조
- 응답 규격
- 에러 처리
- 공통 유틸
- 표준명

### Replace

- 직접 DOM 조작 위주의 레거시 화면 일부
- 중복 JS
- 구조 오염 패키지
- 저장소 산출물 오염

## 5. 차세대 포팅 단위 정의

각 도메인은 아래 단위로 전환한다.

1. 레거시 컨트롤러
2. 레거시 템플릿
3. 레거시 정적 JS
4. 서비스
5. Mapper Interface
6. Mapper XML
7. 핵심 테이블
8. 차세대 대상 API/화면

즉 “파일 단위”가 아니라 “도메인 단위”로 옮긴다.

## 6. 결론

첫 번째 실제 업그레이드 대상은 아래로 고정하는 것이 가장 안전하다.

- `로그인/권한`
- `사용자/회사/부서`

이 공통축이 정리되어야 이후 근태/급여/재무/OCR까지 차세대 구조로 안정적으로 옮길 수 있다.
